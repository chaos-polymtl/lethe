30/**
31 * @brief Mechanism that disables contacts computation through mobility status
32 * of cells based on granular temperature.
33 *
34 * The general idea behind the algorithm:
35 *
36 * It uses the granular temperature to determinate if particles in a cell are
37 * sufficiently mobile that the contact forces are worth computation.
38 * A cell having a granular temperature under a threshold (default is 1e-4) will
39 * have an inactive or advected status which makes them rejected in the broad
40 * search step. There is also an extra layer of mobile cells around the current
41 * mobile cells to propagate the motion of particles. Additionally, there is
42 * another layer of cells around those mobile cells that are flagged as
43 * static_active or advected_active cells.
44 *
45 * While the inactive status makes the particles static (no further handling),
46 * the advected status will lead to computation and application of a
47 * cell-averaged advected term. This term is computed from the last average
48 * velocity and acceleration of the particles when the cell had a mobile status,
49 * or at the first DEM time step of a CFD iteration in CFD-DEM (ensures an
50 * update of the hydrodynamic forces). Those forces are reapplied after each
51 * DEM time step.
52 *
53 * Particles in cells with an inactive status results in skipping the fine
54 * contact search, the p-p & p-w forces computation and the velocity and
55 * position update. It reduces the computational cost of the simulation when
56 * particles are not moving.
57 *
58 * Particles in cells with an advected status results in skipping the fine
59 * contact search, the p-p & p-w forces computations, but they do have a
60 * velocity and position update through the advection term. This is critical for
61 * CFD-DEM.
62 *
63 * In summary, cells are flagged using a mobility status:
64 * - mobile (all contacts are calculated, same as if the feature is not enabled)
65 * - static_active (particles with low velocity variance next to mobile
66 * particles)
67 * - advected_active (same as above but particles will be advected)
68 * - inactive (particles next to mobile particles that allow one way contact
69 * forces calculation, not computation of collision pair in the cell)
70 * - advected (same as above, but particles will be advected)
71 *
72 * There are some edge cases that need some attention. In these cases, particles
73 * in cells cannot be deactivated:
74 *
75 * 1. Solid fraction
76 * The solid fraction of the cell is under a value (default = 40%): particles
77 * in the cell are not consolidated (packed) and may move freely.
78 *
79 * 2. Empty cell neighbor
80 * Cell having empty cell neighbor: it means that particles are next to a
81 * floating wall/mesh, and they may have a change of contact forces from it if
82 * the wall moves or disappears.
83 *
84 * 3. Additional mobile cell layer
85 * The cell has at least one cell neighbor flagged as mobile from the 3 criteria
86 * mentioned above (granular temperature, solid fractions or floating walls):
87 * this is because motion is badly propagated to particles to the cell around
88 * without this additional "layer" of mobile cells.
89 *
90 * 4. Active cell layer
91 * The cell has at least one cell neighbor flagged as mobile from the previous
92 * criterion (additional layer of mobile cells): again, this is to allow the
93 * motion propagation, but only the particles in contact with particles
94 * from the mobile cells are considered for the contact force calculation but
95 * their position is not computed at the integration step. Those cells are
96 * flagged as active cells. It works with the assignment and verification of the
97 * mobility status at nodes to check the status of the neighboring cells.
98 *
99 * @tparam dim An integer that denotes the number of spatial dimensions.
100 * @tparam PropertiesIndex Index of the properties used within the ParticleHandler.
101 */
102template <int dim, typename PropertiesIndex>

108 /**
109 * Mobility status flags used to identify the status at nodes and the status
110 * of the cell:
111 *
112 * inactive (0)
113 * The motion of particles in the cell is considered as negligible,
114 * particles within this cell are not considered in the contact detection
115 * (rejected at the broad search step), so no force calculation or
116 * integration is applied
117 *
118 * static_active (1)
119 * The motion of particles in the cell is considered as negligible, but
120 * there's at least one neighbor cell that is flagged as mobile, meaning that
121 * particles need to be in contact candidates lists at the broad search step,
122 * particles directly in contact with the mobile cell are also considered in
123 * force calculation, but none of the particles in the cell are integrated.
124 *
125 * advected (2)
126 * The variance of the velocity of the particles in cell is low, suggesting
127 * that contact forces are negligible but particles are advected by other
128 * forces. Note that there are currently no mechanisms that distinct active vs
129 * advected status: advection of particles is a user-defined feature. Advected
130 * and active status are detected in the same way, but are not processed the
131 * same at the velocity integration step. For advected particles, the last
132 * computed cell-averaged velocity and acceleration is used to advect the
133 * particles.
134 *
135 * advected_active (3)
136 * Same as (1), but particles are advected by the last computed
137 * cell-averaged velocity and acceleration.
138 *
139 * mobile (4)
140 * The motion of particles in the cell is significant or there is at least
141 * one cell neighbor that is mobile by criteria (see the
142 * identify_mobility_status() description), particles need to be in contact
143 * candidate lists at the broad search step and particles are treated as
144 * usual (force calculation and integration)
145 *
146 * empty (5)
147 * This status is only used for the node-based mobility status identification,
148 * no cells are flagged as empty, only node can be identified as empty.
149 * Without this identification of the empty cells, we cannot identify the cell
150 * that have an empty neighbor cell, which is critical for simulations using
151 * floating walls or mesh.
152 */
153
154 enum mobility_status : unsigned int
155 {
156 inactive = 0, // used for cells
157 static_active = 1, // used for cells and nodes
158 advected = 2, // used for cells and nodes
159 advected_active = 3, // used for cells and nodes
160 mobile = 4, // used for cells and nodes
161 empty = 5 // used for nodes only
162 };

206 /**
207 * @brief Calculate or update the cell-averaged velocities and accelerations
208 * of the cells. This is used for the advection of particles during
209 * integration and the velocities and accelerations map is fully updated at
210 * least at each CFD time step since the first DEM iteration computes every
211 * contact. This map is also partially updated when cell is mobile in the
212 * Verlet integration (not implemented for other scheme).
213 * Since this step loops over all the particles of mobile cells and new values
214 * of velocities are computed, this helps to have a more accurate value of the

216 * next DEM time steps.
217 *
218 * @param[in] particle_handler The particle handler that contains all the
219 * particles.

232 /**
233 * @brief Identify of the mobility status of each cell through a node-based
234 * identification and check. Only the active and ghost cells are processed.
235 *
236 * The following 4 checks (search loops) are done:
237 *
238 * 1. Check if the cell is empty (n_particle = 0), if so, nodes and cells are
239 * flagged as empty mobility status (5)
240 *
241 * 2. Check if the cell is mobile by criteria (average granular temperature >
242 * threshold, solid fraction < threshold or has at least one empty node from
243 * previous check), if so, nodes are flagged and cells are stored with mobile
244 * mobility status (4)
245 *
246 * 3. Check if the cell is mobile by neighbor (at least a node is flagged as
247 * mobile from previous check), if so, cells are stored in map as mobile
248 * status (4) and nodes that are not mobile are flagged as active (1/3)
249 *
250 * 4. Check if the cell is active (at least a node is flagged as active from
251 * previous check), if so, cells are stored with active status in the map
252 * (1/3)
253 *
254 * The remaining cells are inactive (0)
255 *
256 * @param[in] background_dh The dof handler of the background grid.
257 * @param[in] particle_handler The particle handler that contains all the
258 * particles.
259 * @param[in] n_active_cells The number of active cells in triangulation.
260 * @param[in] mpi_communicator The MPI communicator.
261 */
262 void
263 identify_mobility_status(
264 const DoFHandler<dim> &background_dh,
265 const Particles::ParticleHandler<dim> &particle_handler,
266 const unsigned int n_active_cells,
267 MPI_Comm mpi_communicator);

269 /**
270 * @brief Map the periodic nodes pairs of the triangulation using the
271 * constraints. It allows to compare the mobility status of the nodes with the
272 * status of the periodic node.
273 *
274 * Note: this might not be the efficient way to pair the periodic nodes.
275 *
276 * @param constraints[in] The background constraints of triangulation.
277 */
278 inline void
279 map_periodic_nodes(const AffineConstraints<double> &constraints)
280 {
281 periodic_node_ids.clear();
282
283 IndexSet local_lines = constraints.get_local_lines();
284 for (auto i : local_lines)
285 {
286 for (auto j : local_lines)
287 {
288 // Map the values of the periodic nodes (with repetition)
289 if (constraints.are_identity_constrained(i, j))
290 {
291 periodic_node_ids.insert(std::make_pair(i, j));
292 }
293 }
294 }
295 }

375 /**
376 * @brief Assign the mobility status to the cell, its nodes, and its periodic
377 * coinciding nodes from the periodic nodes map. If required, this will
378 * prevent overwriting a node status prevailing over the initial one.
379 * For instance, empty status must not be overwritten by mobile status because
380 * all cells with this node will be part of the additional mobile layer and
381 * will wrongly propagate mobility status to the next cells.
382 *
383 * @param[in] cell_id The current cell index.
384 * @param[in] dof_indices The vector of the DoF indices of the cell.
385 * @param[in] cell_status The mobility status of cell to assign.
386 * @param[in] node_status The current mobility status of the node.
387 */
388 inline void
389 assign_mobility_status(
390 unsigned int cell_id,
391 const std::vector<types::global_dof_index> &dof_indices,
392 const int cell_status,
393 const int node_status)
394 {
395 cell_mobility_status.insert({cell_id, cell_status});
396
397 // Assign mobility status to nodes without overwriting empty or mobile nodes
398 // in regard to the case
399 for (auto node_id : dof_indices)
400 {
401 // Prevailing mobility status of the node and assignation
402 int status_assigned = std::max(node_status, mobility_at_nodes(node_id));
403 mobility_at_nodes(node_id) = status_assigned;
404
405 // Check if node has a periodic node and assign the same mobility status
406 // if prevailing on the one on the periodic node
407 auto it = periodic_node_ids.find(node_id);
408 if (it != periodic_node_ids.end())
409 {
410 mobility_at_nodes(it->second) =
411 std::max(status_assigned, mobility_at_nodes(it->second));
412 }
413 }
414 }
